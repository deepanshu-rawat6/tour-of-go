# Go Assembly (Plan 9): Micro-Optimizing the CPU

While Go's compiler is excellent, there are rare cases where you need absolute control over the CPU's registers and instructions. Go uses its own unique dialect of assembly, originally based on the **Plan 9** operating system.

---

## 🏎️ Why Use Assembly?

1.  **SIMD Support**: Using AVX, AVX2, or NEON instructions for vectorized data processing (e.g., image decoding, hashing).
2.  **Runtime Support**: The Go runtime itself (garbage collector, scheduler, `math` package) is heavily written in assembly.
3.  **Low-Level System Access**: Direct manipulation of CPU state or specialized hardware instructions.

---

## 🏗️ The Plan 9 Dialect

Go's assembly is **not** Intel (x86) or AT&T syntax. It's a cross-platform intermediary language that the linker converts into native machine code.

### 🧩 Key Differences
*   **Operand Order**: Source is on the left, destination is on the right (similar to AT&T).
*   **Virtual Registers**: Use `FP` (Frame Pointer), `SB` (Static Base), `SP` (Stack Pointer), and `PC` (Program Counter).
*   **Symbol Names**: Go symbols are prefixed with `·` (middle dot).

### 🛠️ Example: Adding Two Numbers
**Go code (`add.go`):**
```go
package main

func Add(a, b int64) int64
```

**Assembly code (`add_amd64.s`):**
```asm
// func Add(a, b int64) int64
TEXT ·Add(SB), NOSPLIT, $0-24
    MOVQ a+0(FP), AX    // Load 'a' into register AX
    ADDQ b+8(FP), AX    // Add 'b' to AX
    MOVQ AX, ret+16(FP) // Store result in the return slot
    RET
```

---

## ⚠️ The High Cost of Assembly

1.  **Inlining Loss**: Go functions written in assembly **cannot** be inlined by the compiler. This adds call overhead.
2.  **Maintenance Burden**: You must maintain separate files for different architectures (`amd64`, `arm64`, `386`, etc.).
3.  **No Type Safety**: The compiler cannot verify that your assembly code matches your Go function signature.

---

## 🚀 When to Avoid It

*   **Premature Optimization**: Always profile with `pprof` first. Go 1.17+ uses a register-based calling convention that narrowed the performance gap significantly.
*   **Complex Logic**: Keep assembly as simple and leaf-focused as possible.
